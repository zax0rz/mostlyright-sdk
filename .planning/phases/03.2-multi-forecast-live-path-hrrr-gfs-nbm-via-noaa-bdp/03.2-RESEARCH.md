**Researched:** 2026-05-23
**Domain:** Numerical weather prediction (NWP) raw model output ingest — GRIB2 byte-range fetch + cfgrib decode + BallTree station extraction.
**Confidence:** HIGH (lift source is fully implemented in `sprint2/2r-impl-bundle`, all upstream design decisions cite the architecture RFC, version pins verified against PyPI 2026-05-23, NBM v5.0 cutover date verified against NOAA official sources).

## Summary

Phase 3.2 adds a new entity to tradewinds: **raw NWP model output, station-extracted**, sourced directly from NOAA Big Data Program S3 mirrors (HRRR/GFS/NBM Tier-1). The lift source `sprint2/2r-impl-bundle` is a mature 244-commit branch (~21 KLOC) with R8 fix iterations closed by Codex review. Phase 3.2 lifts only the **live-fetch subset**: `.idx` parser → byte-range fetch → cfgrib decode → BallTree extract → return `pd.DataFrame`. **Skipped: R2/ledger writes, poller, backfill, gold materialization, ECMWF Tier-2, bitemporal queries** — all gated on hosted infrastructure tradewinds does not have.

The implementation pattern is **well-trodden**: Herbie (Brian Blaylock's MIT-licensed package at the University of Utah) demonstrates the same pattern for ~20 NWP models. The mostlyright RFC explicitly cites Herbie as **documentation, not a runtime dep** — bucket URLs, `.idx` format quirks, and `pick_points` BallTree-haversine code are all reimplemented in-house against the lighter dep stack.

**Primary recommendation:**
1. **Lift verbatim** the 8 live-fetch files: `_nwp_idx.py`, `_nwp_grids/{hrrr,gfs,nbm}.py`, `nwp_archive.py`, `_forecast_nwp_columns.py`, `_forecast_nwp_schema.py`, `specs/forecast_nwp.json`, `qc/nwp_rules.py` (12 baseline rules + 1 NBM extension), `qc/nwp_engine.py`, plus `_nwp_constants.py` and `exceptions.py` additions (4 new typed exceptions).
2. **Add the `[nwp]` optional extra** to `packages/weather/pyproject.toml` with `cfgrib>=0.9.15`, `xarray>=2024.0`, `scikit-learn>=1.3`. eccodes binary ships transitively via eccodeslib wheels on Linux/macOS, eccodes wheel itself on Windows — **no manual eccodes install required cross-platform**.
3. **Public surface**: `tradewinds.weather.forecast_nwp(station, model, ...)` mirrors the existing AWC/IEM/GHCNh fetcher pattern. Live SDK returns 37-column fixed-shape DataFrame (36 schema + 1 derived `model_kind`). Cache mirrors observation cache layout: `~/.tradewinds/cache/v1/forecast_nwp/{model}/{station}/{YYYY}/{MM}.parquet`.
4. **Phase 2 dependency**: Phase 2 has NOT yet executed. **Plan 3.2 with explicit Phase 2 interface stubs** — exception base class (`TradewindsError`), canonical schema registry (`tradewinds._internal.specs`), and catalog adapter pattern. These stubs can be filled in when Phase 2 lands; the NWP work is otherwise self-contained.

### Pattern 2: `.idx`-Driven Byte-Range Fetch (the 90% byte reduction)

**What:** GRIB2 model output files are large (HRRR sfcf01 = 135 MB). Each contains ~170 records. We only need ~13 (one per variable). The `.idx` companion file (~10 KB) lists each record's byte offset. Parse `.idx` → filter to needed variables → issue HTTP `Range: bytes=START-END` requests → reassemble into a tiny GRIB2 file → decode with cfgrib.

**When to use:** Every NWP fetch. The byte-range fetch is what makes this feasible at scale.

**Example:**
```python
# Sequence per RESEARCH §3.4 + §5.2 (from mostlyright RFC):
# 1. Fetch the .idx companion file (~10 KB)
idx_text = httpx.get(f"{model_url}.idx").text

# 2. Parse + filter to the ~13 records we need (TMP, DPT, RH at 2m, etc.)
records = parse_idx(idx_text)
records = compute_byte_end(records, content_length=head_response.content_length)
filtered = filter_records(records, variable_map=HRRR_VARIABLE_MAP)

# 3. Issue HTTP Range requests for filtered records (~13 MB total instead of 135 MB)
for rec in filtered:
    range_header = f"bytes={rec.byte_offset}-{rec.end_byte}"
    response = httpx.get(model_url, headers={"Range": range_header})
    # Write each record to a one-message-per-file GRIB2
    record_path = tmp_dir / f"{rec.variable}_{rec.level}.grib2"
    record_path.write_bytes(response.content)

# 4. Decode via cfgrib + xarray
ds = xr.open_dataset(record_path, engine="cfgrib")

# 5. Extract at each station's nearest grid cell via cached BallTree
tree, grid_shape = get_balltree(ds)
station_pts = np.array([[STATIONS[c].latitude, STATIONS[c].longitude] for c in stations])
dist_rad, idx_flat = tree.query(np.deg2rad(station_pts), k=1)
iy, ix = np.unravel_index(idx_flat.ravel(), grid_shape)
values = ds[var_name].values[iy, ix]
```

**Source:** RESEARCH §3 + §5 in `mostlyright/sprint2/2r-s0-architecture-rfc:.planning/2r-s0/RESEARCH.md` [VERIFIED: tmp/2r-s0-RESEARCH.md]

### Pattern 3: HEAD-Resolved EOF for Last Record's byte_end (CRITICAL pitfall)

**What:** The `.idx` file lists each record's start byte. The end byte = next record's start − 1. **For the LAST record, there is no "next record"** — naive parsers return `None` or `NaN`. Cfgrib needs a CONCRETE INTEGER end byte (it crashes on `None`).

**Solution:** Issue an HTTP HEAD request to the GRIB2 file once per `(model_run_path, mirror)` to learn `Content-Length`. The last record's `end_byte = content_length - 1`. Cache the HEAD result per `(model_run_path, mirror)` to avoid re-requesting.

**Example:**
```python
# From sprint2/2r-impl-bundle:ingest/sources/nwp_idx.py:compute_byte_end
def compute_byte_end(records: list[IdxRecord], *, content_length: int | None = None):
    for i, rec in enumerate(records):
        if i < len(records) - 1:
            # Non-last record: chain to next start - 1
            end_byte = records[i + 1].byte_offset - 1
        else:
            # Last record: HEAD-based content_length - 1
            end_byte = content_length - 1 if content_length is not None else None
        out.append(IdxRecord(..., end_byte=end_byte))
    return out
```

**Why it matters:** Codex review caught this as P2 issues in iterations R8 + R9 (`codex R8-P2 + R9-P2 fix`). The earlier `Range: bytes={start}-` (start-to-EOF) form conflicted with the sidecar JSON Schema's `byte_range_end: integer` requirement AND the writer's typed signature. **HEAD-based resolution keeps the fetch API, the sidecar schema, and the replay path all using `int` byte ranges end-to-end.**

**Source:** [mostlyright/sprint2/2r-impl-bundle:ingest/sources/nwp_idx.py:compute_byte_end](https://github.com/Tarabcak/mostlyright/blob/sprint2/2r-impl-bundle/ingest/sources/nwp_idx.py) [VERIFIED: codebase grep]

### Pattern 4: BallTree (haversine, k=1) for Station Extraction

**What:** GRIB2 grids come in different projections — HRRR is Lambert Conformal Conic (curvilinear); GFS is regular lat/lon; NBM is Lambert Conformal. For curvilinear grids, `xarray.sel(method="nearest")` doesn't work directly on 2D coordinate dims. The Herbie pattern (lifted into mostlyright): `BallTree(deg2rad(grid_latlon), metric="haversine")` — works for any grid projection without coordinate transforms.

**When to use:** Every NWP fetch's station-extraction step. Build the tree ONCE per grid layout (cached at module level); query for all 20 stations in a single `tree.query()` call.

**Example:**
```python
# Lift verbatim from sprint2/2r-impl-bundle:ingest/sources/_nwp_grids/hrrr.py:get_balltree
from sklearn.neighbors import BallTree

EARTH_RADIUS_KM = 6371.0

_BALLTREE_CACHE: tuple[BallTree, tuple[int, int]] | None = None
_BALLTREE_LOCK = threading.Lock()

def get_balltree(ds: xr.Dataset) -> tuple[BallTree, tuple[int, int]]:
    global _BALLTREE_CACHE
    if _BALLTREE_CACHE is not None:
        return _BALLTREE_CACHE
    with _BALLTREE_LOCK:
        if _BALLTREE_CACHE is not None:  # double-checked locking
            return _BALLTREE_CACHE
        lat2d = ds["latitude"].values
        lon2d = ds["longitude"].values
        # HRRR longitudes are 0..360; tradewinds stations use signed -180..180
        lon2d = np.where(lon2d > 180.0, lon2d - 360.0, lon2d)
        grid_pts = np.column_stack([lat2d.ravel(), lon2d.ravel()])
        tree = BallTree(np.deg2rad(grid_pts), metric="haversine")
        _BALLTREE_CACHE = (tree, lat2d.shape)
        return _BALLTREE_CACHE
```

**Why k=1 (not k=N nearest neighbors)?** For HRRR's 3-km grid, the nearest cell is essentially "at the station" (max ~1.5 km off). Multi-neighbor interpolation adds latency for ~no signal improvement; we record `grid_dist_km` so the quant can audit and re-interpolate if they want. NBM at 13 km grid spacing: nearest ~6.5 km off — same logic applies.

**Source:** RESEARCH §5.2 in mostlyright RFC; Herbie's `pick_points.py` cited verbatim. [VERIFIED: codebase grep]

### Pattern 5: SSRF Defense-in-Depth (3 layers)

**What:** Mirror URLs come from a closed enum. Never construct URLs via string interpolation from user input.

| Layer | Mechanism |
|-------|-----------|
| 1 | JSON Schema `model_run_archive` enum closure (8 values) — anything else fails schema validation |
| 2 | `_MIRROR_URLS_BY_MODEL` hardcoded dict lookup — `KeyError` on unknown mirror |
| 3 | Per-model URL prefix allowlist in `_nwp_grids/{model}.py` (defense-in-depth beyond the SSRF allowlist) |

**Source:** [mostlyright/sprint2/2r-impl-bundle:ingest/sources/nwp_archive.py + _security.py](https://github.com/Tarabcak/mostlyright/blob/sprint2/2r-impl-bundle/ingest/sources/nwp_archive.py) [VERIFIED: codebase grep]

### Anti-Patterns to Avoid

- **Returning empty DataFrames for unimplemented paths.** A user calling `forecast_nwp(model="ecmwf_ifs_hres")` getting an empty DataFrame can't distinguish "no data available" from "ECMWF not yet implemented." **Always raise `NwpModelNotAvailableError`** for reserved models. Codex caught this in mostlyright R21-P2 review.
- **Storing full GRIB bytes in the cache without hash verification.** Bytes can corrupt; replay validation requires sha256. Store `(sha256, byte_range)` in the sidecar; re-fetch on cache miss matches integrity contract.
- **Building the BallTree per-fetch (not module-level).** Tree build is the expensive step (several MB of grid coordinates). Build once per grid layout; query for all stations in a single call.
- **Silently skipping malformed `.idx` lines.** A bad line most likely means the upstream format changed. Raise `ValueError` with the offending text so the parser update is loud rather than silent.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| GRIB2 decoding | Pure-Python GRIB2 parser, even "just the surface fields" | **cfgrib** | NCEP GRIB2 is a complex binary format with packed integer encoding, complex coordinate systems, ensemble products. Full decoder is ~50 KLOC of C. cfgrib wraps ECMWF's ecCodes binary which is the reference implementation. |
| eccodes binary install | Compile eccodes from C source, document `apt install eccodes-bin` / `brew install eccodes` | **eccodeslib wheel (transitive via eccodes 2.43+)** | Cross-platform compile is hard; conda is heavy; manylinux_2_28+ wheels work out-of-the-box. From eccodes 2.43.0+, the wheel installs the binary library via `eccodeslib` (Linux/macOS) or directly bundled (Windows). pip-install-only on Linux/macOS/Windows. |
| Nearest-grid-cell extraction | Custom KDTree on flattened grid + manual haversine | **sklearn.neighbors.BallTree(metric="haversine")** | BallTree-haversine is the canonical pattern (Herbie + every NWP toolchain uses it). scipy.cKDTree uses Euclidean which gives wrong results at high latitudes / dateline. |
| HRRR/GFS/NBM bucket URL templates | Maintain our own URL builder | **Lift Herbie's URL templates verbatim** | Herbie's models/ subdirectory has been NCEP-maintained for years; URLs are stable, format quirks (GFS pre/post 2021-03-23 `atmos/` subdir; NBM `fxx=0` not provided; HRRR cycle-vs-extended-cycle fhr horizons) are all documented. |
| `.idx` parser | Pandas-based DataFrame parse | **Pure-Python regex parser (50 LOC)** | The `.idx` file is small (~10 KB) and has 6 colon-delimited fields per line. Pandas adds startup cost we don't need at the parse layer. |
| Per-model variable maps | Discover variables per-fetch from the .idx | **Hardcode HRRR/GFS/NBM `VARIABLE_MAP` dicts** | The 13 fields we extract (TMP, DPT, RH, UGRD, VGRD, GUST, APCP, PRES, MSLMA/PRMSL, VIS, TCDC, APTMP) are stable across NCEP product versions. Hardcoding means we don't accidentally pull a different variable when upstream adds new fields. |
| BallTree disk cache | Pickle BallTree to disk + lazy-load | **In-process module-level cache** (per mostlyright pattern) | For local-SDK with short-lived Python processes, in-process is fine. Disk-cache adds complexity for marginal benefit. Sprint-N+ improvement if needed. |
| HTTP byte-range Range header | Roll a custom Range header parser/composer | **httpx native `headers={"Range": f"bytes={start}-{end}"}`** | httpx handles this natively. S3 honors `Range: bytes=START-END`. |
| Mirror fallback chain | Generic httpx retry+rotate-host wrapper | **Per-model `_DEFAULT_*_MIRROR_CHAIN` tuple + explicit catch-and-fallback loop** | Mirror values must come from a closed SSRF allowlist (security), not generic fallback. Mostlyright's pattern: explicit `for mirror in chain: try ... except: continue` keeps the security boundary auditable. |

## Common Pitfalls

### Pitfall 1: Forgetting HEAD-resolved EOF for the last `.idx` record

**What goes wrong:** Last record's `end_byte` is `None`; the `httpx.get(url, headers={"Range": f"bytes={start}-None"})` either malforms the header or silently fetches from `start` to actual EOF. Cfgrib then crashes on the truncated GRIB2 record.

**Why it happens:** Naive `compute_byte_end` chain: `end_byte = next.start_byte - 1`. The last record has no next record.

**How to avoid:** Always pass `content_length` from a one-shot HEAD request to `compute_byte_end`. Cache the HEAD result per `(model_run_path, mirror)`.

**Warning signs:** GribIntegrityError on the last variable (`MSLP_PA` or `TCDC` typically); cfgrib stacktrace at `xr.open_dataset` with "unexpected end of message" or "messages out of order."

### Pitfall 2: BallTree built from longitude in degrees, not radians

**What goes wrong:** `BallTree(metric="haversine")` REQUIRES radians input. Passing degrees gives wildly wrong nearest-neighbor results (off by 100s of km).

**Why it happens:** scikit-learn's docs are quiet on this. Easy to miss.

**How to avoid:** Always `np.deg2rad(grid_pts)` before `BallTree(...)`. And `np.deg2rad(station_pts)` before `tree.query(...)`.

**Warning signs:** `grid_dist_km` shows values >> grid spacing (e.g., HRRR 3-km grid showing 10,000+ km distances).

### Pitfall 3: HRRR longitudes 0..360 vs station longitudes -180..180

**What goes wrong:** HRRR's `latitude` and `longitude` coords come from cfgrib as `(y, x)` 2D arrays, with longitudes in the range `0..360`. tradewinds station coords use signed `-180..180`. Without wrap, west-of-Greenwich grid cells end up "360° away" from station coords.

**Why it happens:** Different upstream conventions.

**How to avoid:** Always wrap `lon2d = np.where(lon2d > 180.0, lon2d - 360.0, lon2d)` before building the BallTree.

**Warning signs:** US stations (NYC, LAX, etc.) showing grid_dist_km > 10,000 km (the dateline-wrap distance).

### Pitfall 4: GFS pre/post 2021-03-23 path layout split

**What goes wrong:** GFS v16.0 reorganization on 2021-03-23 added `/atmos/` subdir. Pre-2021-03-23 path is `gfs.YYYYMMDD/HH/gfs.tHHz.pgrb2.0p25.fFFF`; post is `gfs.YYYYMMDD/HH/atmos/gfs.tHHz.pgrb2.0p25.fFFF`.

**Why it happens:** Historical NCEP reorganization.

**How to avoid:** Lift Herbie's date-conditional in `_nwp_grids/gfs.py:gfs_path()` verbatim. The lift source already handles this.

**Warning signs:** Pre-2021-03-23 GFS fetches return 404 from AWS BDP.

### Pitfall 5: NBM v5.0 cutover at 2026-05-05 06:00 UTC

**What goes wrong:** Runs before 2026-05-05 06:00 UTC contain the previous NBM version; runs from 2026-05-05 06:00 UTC onward use NBM v5. Hourly guidance extends from 36 hours to 48 hours for most parameters. Higher-resolution ECMWF + ECMWF Ensemble + GDPS + REPS guidance added. [VERIFIED: NOAA SCN 26-24 + GribStream blog post]

**Why it happens:** Operational upgrade.

**How to avoid:** **The `.idx` format is unchanged across v4 → v5 cutover. The path layout is unchanged. Field availability differs slightly — v5 adds some probabilistic siblings, but the core 13 fields we extract (and the level descriptors) are stable across the cutover. No date-dependent branching needed in the parser.** mostlyright lift includes test fixtures for both sides of the cutover (`nbm_idx_2024_06_15_t12z_f001.idx` and `nbm_idx_2026_06_15_t12z_f001_post_v5.idx`). Lift both fixtures verbatim.

**Warning signs:** None — handled by the parser being format-stable. If new probabilistic fields cause schema validation errors in v0.1.x, treat as expected and either ignore or add to a v0.2 schema expansion.

### Pitfall 6: NBM `fxx=0` not provided

**What goes wrong:** NBM is itself a blend of upstream model outputs. There is no NBM analysis at t=0. Smallest valid fhr is 1. A user passing `forecast_hours=[0]` for NBM gets 404 from AWS.

**Why it happens:** Operational reality of blended models.

**How to avoid:** Lift Herbie's `if self.fxx == 0 or self.fxx is None: self.fxx = 1` discipline. The lift source already handles this in `_nwp_grids/nbm.py`.

**Warning signs:** NBM fetch returns 404 for fxx=0.

### Pitfall 7: cfgrib version pinning vs eccodes ABI breaks

**What goes wrong:** eccodes ships major ABI version (currently 2.47.0). cfgrib pins to a compatible eccodes range. If we pin cfgrib `>=0.9.10` and eccodes ships a 3.0.0 with breaking ABI, cfgrib breaks silently.

**Why it happens:** ECMWF maintains both; transitions are usually announced.

**How to avoid:** Pin cfgrib upper bound: `cfgrib>=0.9.15,<1.0`. Document eccodes/eccodeslib floor as `>=2.43,<3` in the pyproject `[nwp]` extra comment (not as a direct dep — cfgrib pulls eccodes; we don't).

**Warning signs:** Import errors at runtime referencing `eccodes` ABI mismatch; tests fail with `RuntimeError: ECCODES ERROR :  Wrong format type for the grib file`.

### Pitfall 8: ECMWF model values silently returning empty DataFrames (codex R21-P2)

**What goes wrong:** A user calls `forecast_nwp("NYC", model="ecmwf_ifs_hres")`. The model is predeclared in the enum (per ROADMAP success criterion 1) but the live ingest path raises `NotImplementedError` for it. If the wrapper catches the `NotImplementedError` and returns an empty DataFrame, the caller can't distinguish "no data available" from "ECMWF not implemented in this SDK version."

**Why it happens:** Tempting to surface "no data" as empty rather than as an exception.

**How to avoid:** **Raise `NwpModelNotAvailableError`** (subclass of both `TradewindsError` and `ValueError`) with explicit list of available models. mostlyright fixed this in codex R21-P2 review.

**Warning signs:** Tests pass with empty DataFrames for ECMWF; users complain that ECMWF "silently produces no data."

### Example 1: `.idx` parser line regex (50 LOC for the whole parser)

```python
# tradewinds/weather/_fetchers/_nwp_idx.py
# LIFT: mostlyright/sprint2/2r-impl-bundle:ingest/sources/nwp_idx.py @ 8ec6787

import re
from dataclasses import dataclass

_LINE_RE = re.compile(
    r"^(?P<record_no>\d+)"
    r":(?P<byte_start>\d+)"
    r":d=(?P<date_str>\d{10})"
    r":(?P<variable>[^:]+)"
    r":(?P<level>[^:]+)"
    r":(?P<forecast_time>[^:]+)"
    r":?\s*$"
)

@dataclass(frozen=True)
class IdxRecord:
    record_no: int
    byte_offset: int
    date_str: str          # YYYYMMDDHH
    variable: str          # NCEP short name: TMP, DPT, ...
    level: str             # "2 m above ground", "surface", ...
    forecast_time: str     # "1 hour fcst", "0-1 hour acc fcst", ...
    end_byte: int | None = None

def parse_idx(text: str) -> list[IdxRecord]:
    records: list[IdxRecord] = []
    for lineno, raw_line in enumerate(text.splitlines(), 1):
        if not raw_line.strip():
            continue
        m = _LINE_RE.match(raw_line)
        if not m:
            raise ValueError(
                f".idx parse failed at line {lineno}: {raw_line!r}"
            )
        records.append(IdxRecord(
            record_no=int(m.group("record_no")),
            byte_offset=int(m.group("byte_start")),
            date_str=m.group("date_str"),
            variable=m.group("variable"),
            level=m.group("level"),
            forecast_time=m.group("forecast_time"),
        ))
    return records

def compute_byte_end(records: list[IdxRecord], *, content_length: int | None = None) -> list[IdxRecord]:
    if not records:
        return []
    out: list[IdxRecord] = []
    for i, rec in enumerate(records):
        if i < len(records) - 1:
            end_byte = records[i + 1].byte_offset - 1
        else:
            end_byte = content_length - 1 if content_length is not None else None
        out.append(IdxRecord(**{**rec.__dict__, "end_byte": end_byte}))
    return out
```

```python
#         HRRRv1+v2 forecast skill issues warrant 2018-01-01 cutoff
#         for ML training fitness.
#   GFS:  2021-01-01 (first BDP object — RESEARCH §2.2)
#   NBM:  2020-05-18 (first BDP object — RESEARCH §2.3)
NWP_MODEL_ARCHIVE_START_DATES: dict[str, datetime] = {
    "hrrr": datetime(2018, 1, 1, tzinfo=timezone.utc),
    "gfs": datetime(2021, 1, 1, tzinfo=timezone.utc),
    "nbm": datetime(2020, 5, 18, tzinfo=timezone.utc),
}

def _archive_start_check(model: str, requested_dt: datetime) -> None:
    """Raise if requested_dt is before the model's archive start."""
    start = NWP_MODEL_ARCHIVE_START_DATES[model]
    if requested_dt < start:
        raise ValueError(
            f"NWP {model!r} archive starts at {start.isoformat()}; "
            f"requested {requested_dt.isoformat()} is before that."
        )
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Compile eccodes from C source / `apt install eccodes` / `brew install eccodes` | eccodeslib wheel (transitive via eccodes 2.43.0+) | 2024 (eccodes 2.43.0) | **Pip-install-only cross-platform** — no manual binary install on Linux/macOS/Windows. |
| Open-Meteo seamless time series | Raw NWP model output with `issued_at` provenance | mostlyright v0.17.0+ (post-tradewinds parity baseline) | Quants can train on raw HRRR + self-derive features (RH, feels_like, precip probability) — a more valuable training input than MOS-pre-computed values. |
| IEM MOS forecasts (existing in tradewinds v0.14.1) | Sibling NWP entity (separate schema, separate units, separate model enum) | Phase 3.2 (new) | Three forecast entities (IEM MOS + Open-Meteo + raw NWP), three contracts, three SDK methods. IEM MOS stays unchanged. |
| Manual BallTree-haversine setup | Cached at module level (lazy build, double-checked locking) | mostlyright Sprint 2r-s2 | Tree build is several MB of grid coords; once-per-process amortization. |
| Range header `bytes=START-` (start to EOF) | HEAD-resolved `bytes=START-END` | mostlyright codex R8/R9-P2 fix | Concrete int byte ranges throughout — no nullable end_byte in the sidecar schema; no surprise cfgrib crashes. |
| NBM v4.x (until 2026-05-05) | NBM v5.0 (from 2026-05-05 06:00 UTC) | Operational cutover 2026-05-05 | Hourly guidance extends 36→48 hours; new probabilistic products. **No `.idx` format change; no path change; tradewinds parser stable across cutover.** |

**Deprecated/outdated:**
- **Herbie as a runtime dep** — mostlyright explicitly rejected this; tradewinds inherits the rejection. Use Herbie's MIT-licensed source code as documentation for bucket URLs / `.idx` quirks / per-model gotchas, NOT as a transitive dep. (Cited verbatim with permalinks in source comments.)
- **Pre-eccodeslib eccodes** (< 2.43) — required a system C library install. Not in scope. Pin floor to `cfgrib>=0.9.15` to require eccodes 2.43+ transitively.
- **GRIB2 full-file download** — fetching the full 135 MB HRRR sfc per cycle is feasible but wasteful at scale. Always use the `.idx`-driven byte-range pattern.

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | no (public NOAA endpoints, no auth required) | n/a |
| V3 Session Management | no | n/a |
| V4 Access Control | no (data is public-domain weather model output) | n/a |

### Known Threat Patterns for NWP-fetching code

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| SSRF via mirror URL injection | Tampering / Spoofing | (1) Closed enum at JSON Schema layer; (2) hardcoded `_MIRROR_URLS_BY_MODEL` dict; (3) per-model URL prefix allowlist in `_nwp_grids/*.py` |
| Arbitrary-size response (DoS) | DoS | HEAD request to verify `Content-Length` before issuing byte-range Range header. Cap individual `Range: bytes=START-END` width at `< MAX_RECORD_SIZE` (~10 MB per single GRIB record; the entire HRRR sfc file is 135 MB but individual records max ~3 MB). |
| Tampered GRIB2 bytes downstream | Tampering | sha256 verification of fetched bytes against cache-stored hash (when re-fetching for replay). Mismatch raises `GribIntegrityError`. |
| Malformed `.idx` triggers ReDoS | DoS | `_LINE_RE` regex is linear (no nested quantifiers, no alternation); test on adversarial input fixtures. |
| Cache path traversal | Tampering | Existing `assert_path_under(cache_root)` + `validate_icao_for_path` pattern from `tradewinds.weather.cache`. |
| eccodes binary supply-chain | Tampering | Pin floor for cfgrib `>=0.9.15,<1.0`; document eccodes/eccodeslib floor `>=2.43,<3` in pyproject `[nwp]` extra comment. uv lockfile pins exact transitive versions. |

## Open Questions

1. **Where do `tradewinds._internal.specs.forecast_nwp.json` and the schema registry live before Phase 2?**
   - What we know: `_internal/specs/forecast_nwp.json` directory exists; JSON files are already there (observation.json, climate.json, etc.).
   - What's unclear: Whether Phase 3.2 can drop `forecast_nwp.json` in alongside without a registry yet.
   - Recommendation: Drop the file in `_internal/specs/forecast_nwp.json` directly. Hand-validate with `jsonschema.validate(row, json.load(spec_path))` until Phase 2 wraps it in a registry.

2. **Should the QC engine populate `qc_status` at fetch time (Phase 3.2) or via a separate `tradewinds.weather.qc.run_nwp_qc()` callable that users invoke explicitly?**
   - What we know: ROADMAP §3.2 success criterion 4 says "run at fetch time."
   - What's unclear: Whether running QC inline in `forecast_nwp()` adds enough latency to warrant separation.
   - Recommendation: Run inline. The 12 rules are O(N) over rows; latency overhead is <5ms per 100-row response.

3. **What happens when AWS BDP serves a partial GRIB2 file (e.g., S3 mid-write)?**
   - What we know: AWS BDP atomically swaps files in; mid-write shouldn't be reachable for a public-read URL.
   - What's unclear: Whether mid-write states occur for NOMADS (which is NCEP's lower-availability live mirror).
   - Recommendation: HEAD-then-GET pattern catches `Content-Length` shrinking between calls; raise `GribIntegrityError` on size mismatch. Test with simulated mid-write fixture.

4. **Should the `[nwp]` extra also include a `tradewinds-weather[nwp,parquet]` combination?**
   - What we know: `[nwp]` adds cfgrib + xarray + scikit-learn; `[parquet]` (existing) adds pandas.
   - What's unclear: Whether users will commonly want both.
   - Recommendation: Document `pip install tradewinds-weather[nwp,parquet]` as the recommended install for forecast_nwp + DataFrame support.
