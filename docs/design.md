## Problem Statement

User runs ML models against Kalshi prediction markets (NHIGH/NLOW temperature contracts proven first; broader prediction-market vision behind it). User has lived through the exact pain the spec describes: training and live inference operating on mismatched data, leakage that's invisible until live trading loses money, and the inability to debug "is the model wrong or the data wrong." User shipped a working v1 (mostlyright on PyPI) that solves the weather/Kalshi case, but considers it overengineered: too many abstractions for one domain, and significant infrastructure that the MCP + agent toolchain would now handle natively.

mostlyright-mcp v2 is the planned rebuild: tighter core, MCP-native interface, architected so the second vertical (sports, politics, finance) is a catalog addition rather than a rewrite. v1 ships one vertical fully (weather/Kalshi) to prove the architecture.

## Demand Evidence

**Self-demand (validated):** User shipped mostlyright to PyPI specifically because they hit the schema-drift / leakage / "is it the data or the model" pain trading Kalshi weather contracts. The package has training/inference pairs, NWS CLI settlement truth, IEM forecast joins, quarantine, and validation. It works for the user today.

**External demand (weak):** User can name 1-2 specific people who might want this, plus a fuzzy crowd in r/algotrading and Kalshi Discord. No revenue, no paid users, no one outside the user's friend network has actually used mostlyright in production. The "productize for others" thesis is not yet validated.

This gap is the central risk of the entire project. The architecture work is real, but the demand work has not happened.

## Status Quo

- **For the user personally:** mostlyright (their own package). Works. Considered overengineered.
- **For everyone else:** Hand-rolled scripts per market, notebook spaghetti, or nothing. Quants either build internal infra (most don't share it) or scrape ad-hoc per model.
- **Adjacent tools:** General point-in-time stores exist (Feast, Tecton) but are heavyweight for prediction markets, not domain-aware, and don't ship MCP interfaces.

The real competitor for mostlyright-mcp is "the script the trader already wrote that mostly works."

## Target User & Narrowest Wedge

**Target user (year 1):** A trader/quant who:
- Actively models a specific Kalshi or Polymarket vertical (weather first)
- Already writes Python and runs their own backtests
- Has hit temporal leakage at least once
- Uses Claude Code, Cursor, or another agent-capable IDE

The wedge is *not* "all prediction markets." It is *"weather/Kalshi NHIGH/NLOW with temporal correctness built in, exposed both as a Python lib and an MCP server."*

The "Bloomberg terminal for agents" vision (EDGAR, FRED, court filings, eventually processing data not just BYO) is a 5-year horizon, not a v1 product. The bridge from wedge to vision is *adding catalog modules* without touching core — which is exactly what Approach B is architected for.

## Constraints

- Solo builder, must compress 10x vs mostlyright code volume
- Python core (matches existing skills + mostlyright reuse)
- Cannot ship something that re-creates mostlyright's "too many abstractions" problem
- Must be MCP-callable from Claude Code / Cursor on day 1 (the user is the user; this is how the user works)
- Must remain useful to the user even if zero external users adopt it (sunk-cost insurance)

## Premises (all agreed)

1. **Wedge is "temporal-safe schema enforcement + curated catalog of prediction-market data shapes, weather-first."** Not "Bloomberg for agents." That is the vision, not the v1 product.
2. **mostlyright-mcp v2 subsumes mostlyright** rather than paralleling it. mostlyright becomes mostlyright-mcp's first vertical. Two PyPI packages = confused users.
3. **"Compress 10x" beats "platform with 6 categories."** v1 ships ONE vertical fully (weather/Kalshi) before any second vertical exists.
4. **MCP is the primary surface for v1; the Python library is the underlying engine that MCP exposes.** CLI is deferred to v1.1 or later. Both MCP server and Python import paths ship together — they are not two surfaces, they are one engine and one wrapper.
5. **Catalog adapters DO ingest data (they fetch from NOAA, IEM, etc.); the temporal/schema core also accepts BYO data through `Schema.from_dataframe()`.** Both paths exist in v1, intentionally. They are not "different products" — they are the same engine fed from two sources. The "different products" premise was about avoiding a hosted data warehouse business, which is explicitly out of scope.
6. **Need 3 named external users actively using v1 within 60 days of ship**, or the productize thesis is wrong and mostlyright-mcp stays personal infra. Friends-as-favor does not count. "Actively using" = each user has either filed an issue / PR, sent a written feedback message, or replied affirmatively to a direct "did you run it?" check-in.
7. **Source identity is part of the schema contract, not a hidden SDK default.** Same logical data (e.g. METAR for KATL at 14:00 UTC) can come from multiple sources (`iem.archive`, `iem.live`, `awc.live`) and the values can differ — different QC, different corrections handling, different ingest timing. Silent source substitution between training and inference is leakage-class severity (same blast radius as temporal leakage, different mechanism). mostlyright-mcp encodes it from day 1: `source` is a required parameter, schemas carry source provenance, and validators error on source mismatch between train and infer. (Empirical magnitude of cross-source divergence for the `iem.*` ↔ `awc.live` pair is NOT yet measured — see "Open Questions" for the diff job that needs to land before catalog_search publishes warning numbers.)

## Approaches Considered

### Approach A: Mostlyright Reborn (minimal viable, NOT chosen)
Refactor mostlyright in place — extract temporal-safety + schema core, drop FastAPI, add thin MCP wrapper, rename. 1-3 weeks. Lowest risk to user demand validation (fastest to "does anyone else want this?"), highest risk to architecture (inherits mostlyright's "too many abstractions" problem unless ruthlessly cut).

### Approach B: Core + Catalog + MCP (ideal architecture, CHOSEN)
Clean three-layer build:
- `mostlyright_mcp.core` — temporal primitives, schemas, validators, leakage detection
- `mostlyright_mcp.catalog` — pluggable source adapters (weather first)
- `mostlyright_mcp.mcp` — MCP server exposing catalog as knowledge + validation tools

4-6 weeks. Architected so vertical N+1 is a catalog module, not a rewrite. Bet: a clean rebuild is faster than refactoring mostlyright into the same shape, and the MCP surface on day 1 is load-bearing for the positioning.

### Approach C: Just the Primitive (lateral, NOT chosen)
Ship mostlyright-mcp as a pure temporal-safety + schema-validation library. No catalog in core. mostlyright becomes the first example consumer. "Pydantic for temporal data" positioning. Smallest possible surface, but loses the domain-awareness moat and competes with Feast/Tecton.

## Recommended Approach

**Approach B: Core + Catalog + MCP.**

### Rationale (user's call, against my A recommendation)
The user picked B with reasoning. Two points where B is genuinely stronger than A:
1. **Refactor-to-clean is often harder than clean-from-scratch.** Mostlyright accumulated abstractions across iterations. Extracting the load-bearing pieces while preserving behavior is more painful than reimplementing the load-bearing pieces from intent.
2. **MCP on day 1 is positioning load-bearing.** "MCP-native" is the differentiation vs Feast/Tecton/internal scripts. Adding MCP later as a wrapper makes it feel like an afterthought.

### Where the risk lives
The 4-6 week timeline before any external user touches v1 is the real cost. Premise P6 (3 external users in 60 days of ship) is the test. Mitigations baked into the plan:
- Ship a public alpha at week 3 (`mostlyright-mcp 0.1.0a1` on PyPI) even if the catalog is incomplete. Get the first external installer before the core is "done."
- Start posting build-in-public weekly to /r/algotrading and Kalshi Discord from week 1. Distribution work runs parallel to build, not after.
- The 2 known users get told the dates and asked to commit to trying v1 by week 6.

### Definitions (load-bearing — must be precise before code)

- **`event_time`** — when the event being recorded actually occurred (e.g., the timestamp of a temperature observation, the date a Kalshi contract settles).
- **`knowledge_time`** — *when this row first became publicly knowable.* v0.1 has no continuous ingester, so this is computed at adapter time by one of three rules, depending on data type:
  - **Observations (IEM/AWC, archive and live):** `event_time + per-source publication lag constant`. The constant is a class-level attribute on each adapter (e.g. `IEM_METAR_LAG = timedelta(minutes=15)`), documented and auditable. METAR routine cycle is HH:51 or HH:55 and hits IEM/AWC within minutes; the lag constant is a conservative bound on that distribution. Same rule for both `*.archive` and `*.live` since there's no service stamping arrival time — pulling "right now" returns rows that were already publicly knowable per the lag bound. SPECI (off-cycle METARs) use the same rule.
  - **Forecasts (IEM MOS):** `knowledge_time = issued_at`, read from the source's `runtime` field per record. Source-served, no heuristic. mostlyright already normalizes this at [_forecast_parse.py:232](../../../Documents/GitHub/mostlyright/src/mostlyright/_forecast_parse.py:232).
  - **Settlement (NWS CLI):** `knowledge_time = product release timestamp` parsed from the CLI product header. Source-served, no heuristic.
  - Worked examples:
    - IEM observation for `event_time=14:51 UTC`, lag constant 15min → `knowledge_time=15:06 UTC` regardless of when the user actually fetched (archive or live).
    - IEM forecast with `runtime=12:00 UTC`, valid_at 18:00 UTC → `event_time=18:00, knowledge_time=12:00`.
    - NWS CLI for 2026-05-21, product released 06:32 UTC on 2026-05-22 → `event_time=2026-05-21, knowledge_time=2026-05-22T06:32Z`.
- **Point-in-time invariant:** at any `as_of=T`, only rows where `knowledge_time <= T` are visible. This is the only invariant the temporal-window layer is responsible for.
- **`source`** — the *identity of the data provider + endpoint* that produced the row. Format: `"<source>.<endpoint>"`. Examples: `iem.archive`, `iem.live`, `awc.live`, `cli.archive`. Source is a first-class column on every row and a required parameter on every fetch call. Reason: same logical observation can come from multiple sources with materially different values (different QC, timing, corrections); silent substitution between training and inference is leakage-class severity. See P7.
- **Source-identity invariant:** schemas record which source the training data came from. `validate_dataframe(df, schema_id)` errors if `df` carries a different source than the schema was registered with, unless the caller explicitly opts out via `allow_source_drift="<reason string>"` (per Amendments §J — boolean form is no longer accepted). This makes the train/infer source contract enforceable in code.

### SDK contract (Python import path)

The SDK is the same engine MCP exposes. Public methods on `MostlyRightMCPClient`:

```python
client.observations(
    station: str,                  # e.g. "KATL"
    source: str,                   # REQUIRED. e.g. "iem.archive", "awc.live"
    obs_type: str,                 # e.g. "METAR"
    from_date: str,                # ISO date
    to_date: str,                  # ISO date
    units: str = "metric",
    format: str = "dataframe",     # "dataframe" | "json" | "parquet" | "toon" | "csv"
) -> pd.DataFrame | list[dict] | bytes | str

client.climate(
    station: str,
    source: str = "cli.archive",   # only valid value for v0.1 (settlement is always historical); explicit anyway
    from_date: str,
    to_date: str,
    units: str = "metric",
    format: str = "dataframe",
) -> ...

client.forecasts(
    station: str,
    source: str,                   # e.g. "iem.archive", "iem.live"
    from_date: str,
    to_date: str,
    format: str = "dataframe",
) -> ...

client.pull_pairs(
    contract: str,                 # e.g. "KXHIGHNYC"
    sources: dict[str, str],       # REQUIRED. e.g. {"observations": "iem.archive", "forecasts": "iem.archive", "settlement": "cli.archive"}
    from_date: str,
    to_date: str,
    include_forecast: bool = True,
    format: str = "dataframe",
) -> ...
```

**Source parameter rules:**
- Required on every fetch method. No implicit default for `observations`/`forecasts`. `climate` has only one valid source in v0.1 so it's defaulted, but still settable.
- Format: `"<source>.<endpoint>"` single string. Endpoints: `archive` | `live`. Adding new endpoints later (e.g. `iem.mos`) doesn't break the API shape.
- Silent fallback between sources is forbidden. If `iem.live` errors, the SDK raises `SourceUnavailableError` — it does NOT try `awc.live` or `iem.archive`. Falling back across sources would silently violate the source-identity invariant.

**Format parameter rules:**
- Default `"dataframe"` for SDK calls (Python users want pandas, matches mostlyright today).
- Same data + schema + temporal stamps across all formats. Only the serialization changes.
- TOON is the token-efficient text format reused from mostlyright's `_toon.py` / `_toon_list_codec.py`.

**Exceptions:**
- `MostlyRightMCPError` — base class
- `SourceUnavailableError` — fetch failed for the requested source (no fallback)
- `SchemaValidationError` — row(s) violated schema contract
- `SourceMismatchError` — `validate_dataframe` saw a different source than the schema was registered with
- `LeakageError` — `LeakageDetector` found a row with `knowledge_time > as_of`

### Layer responsibilities

**`mostlyright_mcp.core`** (Weeks 1-2):
- `TimePoint` — wraps timestamps with explicit timezone + DST handling
- `KnowledgeView` — query interface that filters rows by `knowledge_time <= as_of`. Structurally prevents leakage when used.
- `Schema` — declarative schema with `event_time`, `knowledge_time`, and `source` columns required. Records training-time source provenance. Includes `Schema.from_dataframe(df, ...)` for BYO data.
- `Validator` — runs schema contract checks, quarantines bad rows, surfaces lineage. Enforces source-identity invariant (errors on source mismatch).
- `LeakageDetector` — for user-built training sets that did NOT come through `KnowledgeView` (e.g., user joined external data manually): given a training set + a knowledge_time, asserts no row violates point-in-time discipline. This is the audit tool, not the structural guard.
- `formats/` — serializers for `dataframe`, `json`, `parquet`, `toon`, `csv`. TOON lifted from mostlyright's `_toon.py` and `_toon_list_codec.py`.

**Test bar for core:** Property-based tests (Hypothesis) for `KnowledgeView` and `LeakageDetector` covering temporal invariants. Property tests for source-identity invariant (random source pairings, schema mismatches must always error). **≥90% branch coverage on `core/` (not line coverage — see Amendments §H for the explicit negative-test matrix).** Schema/Validator have unit tests for each documented edge case (missing event_time, naive timestamps, DST boundaries, source mismatch with and without `allow_source_drift`). Format roundtrip tests: same DataFrame through each format and back is identical (lossy formats document their losses per §I).

**`mostlyright_mcp.catalog`** (Weeks 3-4 for weather):

Source set is determined by what `mostly-light` actually consumes today (so the integration parity criterion below can pass). The real sources, verified in `mostlyright/src/mostlyright/weather/`, are:

| Adapter | Source IDs | Provides | Why load-bearing for v0.1 |
|---------|-----------|----------|---------------------------|
| `catalog.weather.iem` | `iem.archive`, `iem.live` | Observations + forecasts | mostly-light's `client.observations(...)` (→ `iem.archive` or `iem.live`) and `WeatherLive` (→ `iem.live`) both route here |
| `catalog.weather.awc` | `awc.live` (and `awc.archive` if API supports) | METAR JSON observations | Matches mostly-light's local `metar_parser.py` column contract; eliminates the duplication smell |
| `catalog.weather.cli` | `cli.archive` | Daily settlement truth (high/low) | mostly-light's `client.climate(...)`; non-negotiable — it's the label for Kalshi NHIGH/NLOW. Settlement is always historical so only `archive` endpoint exists. |

Each adapter publishes its `source` IDs as a class-level list (`SUPPORTED_SOURCES = ["iem.archive", "iem.live"]`), and the dispatcher routes calls based on the `source` parameter. Catalog `feature_catalog()` returns the full source matrix.

Plus `catalog.weather.kalshi_nhigh` — the schema definition for NHIGH/NLOW contracts (the consumer of the above). Not a fetch adapter; a contract spec that maps settlement source → contract resolution.

**v0.1.0 ships all three fetch adapters + the Kalshi schema.** This is more than the "2 adapters" the prior draft of this doc said. Reason: the prior draft was wrong — it assumed Open-Meteo, which was removed from mostlyright in v0.15.0 and is not in mostly-light's call path.

**Deferred to v0.2:** `catalog.weather.ghcnh` (GHCN-hourly historical) — only add if `kxhigh` or other in-scope strategies actually depend on it. Currently unverified.

Each adapter is a small module: declares schema, fetches data, returns normalized observations stamped with `knowledge_time`.

**Test bar for catalog:** Each adapter has a recorded-fixture integration test (saved HTTP responses replayed). Adapter "working" = pulls last 30 days for ≥3 stations (where applicable), all rows pass schema validation, knowledge_time matches the documented rule (`event_time + per-source lag` for observations, `issued_at` for forecasts, product-release time for CLI settlement).

**`mostlyright_mcp.mcp`** (Weeks 5-6):

MCP server exposing exactly THREE tools in v0.1.0:

- **`catalog_search(market: str, contract_type: str) → JSON`** — returns the source matrix for this contract family, including:
  - Available sources per data type (observations / forecasts / settlement)
  - **Validated combinations** (which `{observations, forecasts, settlement}` source tuples have been backtested cleanly together — e.g. `{obs: "iem.archive", fcst: "iem.archive", settle: "cli.archive"}`)
  - **Known-bad combinations + warnings**, with numbers backed by an actual cross-source diff job (NOT hypothetical). The shape of a warning is: "do NOT mix `iem.archive` obs with `awc.live` obs in the same training set — measured mean offset X°F, p99 abs diff Y°F over <station set, date range>." Until the diff job runs (see Open Question 4), `catalog_search` returns warnings with `"status": "unmeasured"` and the numeric fields null, rather than fabricated values. Ship gate does NOT require measured numbers; it requires the placeholder mechanism to be honest about what's known.
  - Schema docs, gotchas, timezone notes
  - Default response format: JSON (small metadata, fits comfortably in agent context).

- **`pull_pairs(contract: str, sources: dict[str,str], from_date: str, to_date: str, include_forecast: bool = True, format: str = "toon") → bytes|str`** — return training pairs (replicates mostlyright `pairs()`).
  - `sources` is REQUIRED: `{"observations": "iem.archive", "forecasts": "iem.archive", "settlement": "cli.archive"}`. No silent default. Agent reads from `catalog_search` first if unsure.
  - `format` (optional, default `"toon"`): `"toon"` for agent-context consumption (token-efficient), `"parquet"` for large results agent will persist/forward, `"json"` for small results or debugging.
  - Return includes source provenance per column. Schema is registered server-side; the response includes the registered `schema_id` for later validation.

- **`validate_dataframe(data: bytes|str, schema_id: str, format: str = "toon", allow_source_drift: str | None = None) → JSON`** — runs schema contract checks including source-identity invariant. (Boolean form removed per Amendments §J; pass a reason string to opt out.)
  - Wire format: client serializes DataFrame to the chosen format, sends, server returns JSON report with violations + quarantine summary + source-mismatch flags.
  - If `allow_source_drift` is `None` (default, per §J — the legacy `=False` form is removed) and the data's source differs from the schema's registered source: error with both sources named and a pointer to the relevant `catalog_search` "known-bad combinations" warning.

DEFERRED: `as_of_query(source, query, knowledge_time)` — generic query passthrough is significant complexity (parser, DuckDB integration, injection surface). Ship only when a named user requests it; `pull_pairs` already covers the common path.

Knowledge resources: 1-page markdown per adapter (schema, gotchas, timezone notes, source-pairing rules), served as MCP resources. Cap at 1 page each; do not gate ship on docs completeness.

**Test bar for MCP:** Hello-world MCP spike (see Week 0 below) recalibrates this estimate. Integration tests: `mostlyright-mcp-server` spawned as subprocess, each tool invoked over JSON-RPC, response shape matches contract. Round-trip test: `pull_pairs(..., format="X")` → `validate_dataframe(..., format="X")` succeeds for every supported format. Source-mismatch test: validate fails with clear error when sources differ across train and infer.

### Data persistence

**v0.1 — thin SDK, no infra:**
- v0.1 stores nothing the user doesn't ask for. No background indexer, no opinionated cache directory, no hosted service.
- `pull_pairs` and adapter calls hit public APIs directly (IEM, AWC, NWS CLI) and return data to the caller; the caller decides whether to persist (parquet, DuckDB, in-memory).
- Optional convenience: `pull_pairs(..., cache_dir=...)` writes to user-specified parquet. Off by default.

**v0.2 — transparent read-through cache (deferred, not in ship gate):**
- Hosted R2 cache over the same public-API source-of-record. Cache rows are **semantically equivalent under the canonical schema** — same `source` ID (`iem.archive`, etc.), same `event_time` and column values per the schema's contract test, same `knowledge_time` rule. Not byte-identical to source bytes (source returns CSV/JSON, cache returns normalized parquet); see Amendments §K for the precise definition. The cache is a speedup, not a different source ID.
- Pre-warm strategy for v0.2: manually curated set of Kalshi NHIGH/NLOW contract stations × full available history (~30 stations × ~30 years METAR ≈ **2.7-3.6 GB** ZSTD-compressed parquet total — see Amendments §K for the math; the original "100MB" estimate in earlier drafts was off by ~30x). Still trivial R2 cost (<$0.07/month at $0.015/GB-month). No read-through cache populated from user telemetry until usage exists to learn from.
- **What does NOT get cached:** live endpoints (`*.live`) by definition return current data; archive data within the last ~30 days may still receive METAR amendments and stays direct-from-source. Cache covers the stable tail only — effective TTL "forever" for old data, "never" for recent.
- **Redistribution constraint:** only sources whose terms permit redistribution may enter the cache. NOAA/IEM/AWC/NWS CLI are US government work / public domain — eligible. Open-Meteo and any future sources require per-source legal review before cache eligibility (Open-Meteo's removal in mostlyright v0.15.0 was for exactly this reason).
- **If cache ever transforms data beyond pure pass-through** (re-stitching, additional QC, cross-source enrichment), it gets its own source ID (e.g. `iem.archive.curated`) — never silently overloads the public-API source ID. Source-identity invariant remains enforceable.
- The hosted cache is the natural commercial layer if demand validation passes the 60-day gate; v0.1 ships without it and remains useful even if the cache is never built.

## Open Questions

1. **What happens to mostlyright on PyPI?** Decision: deprecate with a redirect message in the README once mostlyright-mcp 0.1.0 ships with feature parity (week 6-7). Existing mostlyright installs continue to work; the package is frozen, not deleted. Sunset notice in mostlyright's README points to mostlyright-mcp.
2. **Authentication for hosted catalog knowledge.** Not needed in v1. v1 is local-only, no user-specific state. Revisit only if a hosted layer is built post-validation.
3. **The 2 named users — who are they specifically, and have they committed to trying v1?** Not answered in this session. See "The Assignment" — this is a Week 0 gate, not parallel work.
4. **Cross-source divergence — empirical magnitudes.** The source-identity invariant is structurally defensible (silent substitution between any two endpoints is leakage class), but the `catalog_search` warning numbers that justify it to users need actual measurement, not stories. Diff job to run before v0.1.1: pull `iem.archive` vs `awc.live` for the same 3–5 stations × 90-day window, compute mean / p50 / p95 / p99 abs diff on each shared column. Same for the `iem.archive` ↔ `iem.live` overlap window. Write results into `catalog_search`'s warnings. **Not a v0.1 ship blocker** — v0.1 ships with `"status": "unmeasured"` placeholders. v0.1.1 follow-up.
5. **Per-source publication lag constants — calibration.** The lag heuristic (`event_time + IEM_METAR_LAG`) needs an empirical value per source/data type. ### B. Source-identity invariant — third leakage class (archive amendments)

Same `source` ID does not imply same bytes for the same logical row. `iem.archive` can return amended observations or revised forecasts between two pulls of the same `(station, event_time)`. This is a leakage class the original invariant misses.

**Mitigation, in this order:**

1. Every row carries `retrieved_at` (wall-clock timestamp of the pull that produced the row). Schema records `retrieved_at_min` and `retrieved_at_max` on registration.
2. `validate_dataframe(df, schema_id, allow_temporal_drift: str | None = None)` errors if any row's `retrieved_at` is outside the schema's registered range AND its `event_time` is within `retrieved_at - 30 days` (the volatile window). Older data is treated as stable.
3. For `iem.archive` specifically: schemas registered with rows whose `event_time` falls within the 30-day volatile window are flagged with `volatility_warning=True` in metadata, and `catalog_search` surfaces this.

Source identity is therefore the tuple `(source, retrieved_at_range)` for the volatile slice and just `source` for the stable tail.

### C. `pull_pairs` source identity — per-role

`pull_pairs` joins three data types; the result carries three source columns and three retrieved-at columns:

- `obs_source`, `obs_retrieved_at`
- `fcst_source`, `fcst_retrieved_at`
- `settle_source`, `settle_retrieved_at`

`validate_dataframe` against a paired schema enforces each role's source identity independently. A typo (`iem.live` vs `iem.archive`) on any single role errors out with `SourceMismatchError` naming the offending role.

### D. SDK exception payloads (structured)

All exceptions subclass `MostlyRightMCPError`. Attributes are accessible in Python and serialized in `error.data` for MCP JSON-RPC responses.

```python
class MostlyRightMCPError(Exception):
    error_code: str            # stable enum, e.g. "SOURCE_UNAVAILABLE"
    source: str | None         # source ID involved
    request_id: str | None     # MCP request ID if applicable

class SourceUnavailableError(MostlyRightMCPError):
    http_status: int | None
    retryable: bool
    retry_after_s: float | None
    underlying: str            # repr of underlying exception
    url: str | None            # request URL (with secrets redacted)

class SchemaValidationError(MostlyRightMCPError):
    schema_id: str
    violations: list[dict]     # row-level: {row_idx, column, expected, actual, rule}

    sample_violations: list[dict]  # capped at 10 for MCP wire serialization

class SourceMismatchError(MostlyRightMCPError):

    data_source: str

    violating_count: int
```

### E. MCP tool wire formats and response shapes

### I. TOON format losses (documented)

TOON lifted from mostlyright loses:

- pandas `Categorical` dtype → string (category metadata dropped)
- `Timestamp[ns, <tz>]` precision below microsecond (TOON serializes µs, not ns)
- Mixed-type `object` columns containing dict/list values: nested values dropped (TOON is flat)
- `Int64` nullable integer columns: serialized as float with NaN for missing (precision loss above 2^53, and dtype changes on roundtrip)

Roundtrip test asserts these losses are **deterministic** — same input always yields same lossy output. Lossy columns flagged in TOON header metadata.

### J. `allow_source_drift` safeguard

Signature changed from `allow_source_drift: bool = False` to `allow_source_drift: str | None = None`. To opt out of the invariant, caller must pass a non-empty reason string (logged to schema audit trail via `Schema.audit_log()`). `True`/`False` no longer accepted. MCP wire form: `"allow_source_drift": "switched to backup endpoint during outage 2026-05-21"` or omitted.

### K. v0.2 cache storage estimate — corrected

Original "~100MB parquet for 30 stations × 30 years METAR" is off by ~30x. Realistic: IEM METAR for one major ASOS station is ~8,800 routine + ~500 SPECI rows/year, ~12 typed columns post-normalization. ## Amendments — Second Review Pass (2026-05-21)

The first-pass amendments above introduced new issues. Joint second-pass review (Claude + Codex) surfaced ~10 P1/P2 problems with the first-pass spec. Resolved below. Where this section conflicts with the first-pass Amendments or the original sections, **this section is authoritative.**

### P. `retrieved_at` validation — fix live-inference rejection

The first-pass §B as written would break routine live inference: a training schema registered last year has an old `retrieved_at_min/max`; today's live inference row has `retrieved_at = now`, which is outside the registered range AND its `event_time` may fall within the 30-day volatile window of `now`. The naive rule errors on every inference call.

**Corrected semantics — two separate checks, each opt-in or default-on as appropriate:**

1. **Source/schema compatibility (default-on, the leakage-class invariant from §B):** `validate_dataframe` confirms `df.source` matches the schema's registered source. Live inference rows ALWAYS pass this regardless of when they were pulled.

2. **Dataset reproducibility audit (opt-in, separate API):** when *reproducing* a historical training run, the caller passes `assert_retrieved_at_range=(min, max)` to `validate_dataframe`. If any row's `retrieved_at` is outside the asserted range AND its `event_time` is within `(retrieved_at - 30 days)` (the volatile window), error with `TemporalDriftError` carrying the offending rows. Default is `None` = no check. This is for backtest replay correctness, not the every-call invariant.

3. **Volatility metadata (advisory):** schemas registered against rows whose `event_time` falls within the 30-day volatile window of the original pull are flagged `volatility_warning=True` in metadata; `catalog_search` surfaces it. The flag is informational — it does NOT cause `validate_dataframe` to error by itself.

### Q. MCP binary payloads — JSON-RPC encoding

The first-pass §E said inline payloads could be `bytes` (parquet) or `str` (toon/json/csv). JSON-RPC cannot carry arbitrary bytes. Corrected:

**Inline payloads (≤ 5 MB serialized):**
- String formats (toon/json/csv): inline as `str` directly in JSON-RPC params/result.
- Binary formats (parquet): base64-encoded inside a JSON envelope:
  ```json
  {
    "format": "parquet",
    "payload_base64": "...",
    "size_bytes": N,
    "schema_id": "schema.observation.v1",
    "request_id": "..."
  }
  ```
  base64 inflates size ~33%; the 5 MB cap is on the base64-encoded payload, so ~3.75 MB of raw parquet inline.

**File-path mode (> 5 MB):**
```json
{
  "format": "parquet",
  "path": "/Users/.../mostlyright-mcp/abc123.parquet",
  "size_bytes": N,
  "schema_id": "...",
  "ttl_s": 3600,
  "request_id": "..."
}
```
`request_id` added for log correlation across the response and the eventual cleanup. Path is absolute, normalized via `Path.resolve(strict=True)`, and MUST lie inside `$MOSTLYRIGHT_MCP_TMPDIR` (realpath containment check). Symlinks are rejected (the realpath of the file must equal the original path).

**Reaper:** the MCP server runs a background sweeper on startup and every 60 seconds that removes files in `$MOSTLYRIGHT_MCP_TMPDIR` whose mtime is older than `max(ttl_s, 3600)`. The sweeper runs even if no clients are connected (it's a server housekeeping task, not a per-request action).

**Symlink/race protection:**
- Files are created with `O_CREAT | O_EXCL | O_NOFOLLOW` semantics where supported (POSIX). On platforms without `O_NOFOLLOW`, the server explicitly stat-checks and refuses symlinks.
- tmp dir is created with mode `0o700` and verified owner-only on each startup.
- Reads via the path returned in file-path mode go through the same realpath containment check.

### R. Exception hierarchy — additions from §E and §B

The first-pass §D missed two error classes referenced elsewhere in the amendments. Adding them, and standardizing the role-name vocabulary at the same time.

```python
class TemporalDriftError(MostlyRightMCPError):
    """Raised by reproducibility audit (§P) when retrieved_at falls outside
    an asserted range for rows in the volatile window."""
    schema_id: str
    asserted_range: tuple[str, str]   # ISO timestamps
    violating_rows: int               # count
    sample_violations: list[dict]     # capped at 10, JSON-safe (see §S)

class PayloadTooLargeError(MostlyRightMCPError):
    """Raised by MCP server when an inline payload exceeds the cap and
    the client did not use file-path mode."""
    declared_size: int
    limit: int
    accepted_modes: list[str]         # e.g. ["file-path"]
```

**Role-name vocabulary — standardized:** the three `pull_pairs` roles are spelled `"observations"`, `"forecasts"`, `"settlement"` everywhere (matching the source-tuple keys). The column-prefix abbreviations `obs_`, `fcst_`, `settle_` are ONLY used in column names (where short prefixes are preferred for readability). `SourceMismatchError.role` carries the long name (`"observations"` etc.), not the prefix.

### S. JSON serialization coercion for `*Error.violations` / `sample_violations`

The violation dicts will contain pandas types (`Timestamp`, `NaT`, numpy floats with `NaN`, `Int64` with `<NA>`). These are not JSON-serializable by default. Specification:

- Before serialization, all violation dicts pass through a JSON-safe encoder:
  - `pd.Timestamp` → ISO 8601 string (UTC)
  - `NaN`, `NaT`, `pd.NA`, `None` → JSON `null`
  - numpy scalar types → native Python (`int`, `float`, `bool`)
  - ndarray slices → `list`
  - any value still not JSON-serializable after this pass → string repr with `"_repr_only": true` flag
- The encoder is applied by the SDK before the exception is raised. Receivers of `error.data` over MCP can trust the payload is JSON.

`violations` (full list) is capped at 10,000 entries; surplus violations cause the SDK to write the full list to a tmp file (using the §Q file-path mode) and put `{"violations_overflow_path": "...", "total_violations": N}` in the exception data. `sample_violations` (≤10 entries) is always inline.

### U. Settlement timezone — local climate day

The first-pass §A treated `event_time = observation_date at 00:00 UTC`. That's wrong for climate-day contracts: a CLI for "2026-05-21 KORD" describes Chicago local calendar day, not a UTC day. DST further complicates UTC mapping.

**Corrected:** the settlement schema carries:
- `observation_date` (date) — the local climate day per NWS convention (no timezone applied).
- `station_tz` (string) — the IANA timezone for the station (`"America/Chicago"` for KORD), looked up from a static table per ICAO code. Added as a schema column.

### V. Quarantine vs schema-valid rows

**Specification:**
